# 奖励函数设计——优化的目标从哪来？

## 本节导读

**核心内容**

- 掌握奖励函数如何把任务目标转化为可优化的标量信号。
- 理解稀疏奖励、奖励塑形、奖励作弊、内在奖励各自解决什么问题、带来什么风险。
- 学会用势函数塑形在不改变最优策略的前提下增强学习信号。

**核心公式**

$$R(s,a) = \begin{cases} +1 & \text{达到目标} \\ 0 & \text{其他} \\ -1 & \text{失败} \end{cases}$$

$$F(s,a,s') = \gamma\Phi(s')-\Phi(s)$$

$$r_t^{intrinsic} = \|f(s_t,a_t)-s_{t+1}\|^2$$

$$r_t^{total} = r_t^{extrinsic}+\beta r_t^{intrinsic}$$

前两节我们走过了两条路线：路线一学 $Q(s,a)$ 给动作打分，路线二直接优化策略 $J(\theta)$。两条路线的优化方法截然不同，但它们都在优化同一件事——**累积奖励的期望**。

这就引出了一个被我们默认到现在却从未审视的问题：**奖励函数 $R(s,a)$ 从哪来？谁设计的？设计得好不好？**

> 如果优化方法是引擎，奖励函数就是方向盘。引擎再强，方向盘指错了方向，车只会越开越偏。

本节将讨论奖励函数设计中的核心问题——它是 RL 工程实践中最被低估、也最容易出问题的环节。

## 奖励假说：一切目标都是标量奖励？

在 MDP 的定义中，奖励 $R(s,a)$ 是一个标量——一个实数。Sutton 在 2018 年提出了著名的**奖励假说（Reward Hypothesis）**：

> _所有我们所说的目标和目的都可以被理解为：最大化接收到的标量奖励信号的累积和的期望。_

这个假说声称，无论你想要什么——赢棋、赚钱、学知识、获得幸福——理论上都可以写成一个标量奖励函数，然后用 RL 来优化。

这听起来很强大，但也引发了很多争议：

- **支持者**认为，既然任何偏好关系都可以编码为效用函数（von Neumann-Morgenstern 定理），那标量奖励在理论上足够通用。
- **质疑者**指出，现实中的目标往往是多维的、相互矛盾的（比如"既要快又要安全"），压成一个标量会丢失信息；而且人类自己都说不清楚自己想要什么，又怎么能设计出正确的奖励函数？

这个争议在近年来变得愈发尖锐。在实际的 RL 系统中，我们经常面对**多目标优化**的问题：自动驾驶既要到达目的地，又要保证安全，还要节省能源。把这三个目标压成 $R = \alpha \cdot R_{\text{speed}} + \beta \cdot R_{\text{safety}} + \gamma \cdot R_{\text{energy}}$ 看似可行，但 $\alpha, \beta, \gamma$ 怎么取？不同的权重组合会导致完全不同的行为——而"正确的"权重组合本身就是一个需要人类反复调试的超参数。更棘手的是**安全约束**：有些行为是不可接受的（比如自动驾驶闯红灯），你不能用"扣几分"来处理，而需要硬性约束（constrained RL / safe RL）。这些挑战表明，奖励假说虽然在理论上优雅，但实践中设计出一个"正确"的标量奖励函数远比看起来困难。

无论你站在哪一边，奖励假说提醒我们一件事：**RL 的性能上限取决于奖励函数的质量。** 再好的算法，喂给它一个糟糕的奖励函数，也学不出好策略。

## 稀疏奖励：信号太弱，学不动

最自然的奖励设计方式往往是最"懒"的：赢了给 +1，输了给 -1，其余时候给 0。

$$R(s,a) = \begin{cases} +1 & \text{达到目标} \\ 0 & \text{其他} \\ -1 & \text{失败} \end{cases}$$

这种设计在理论上完全正确——最优策略确实会最大化这个期望。但在实践中，**信号太弱了**。智能体在随机探索时几乎不可能碰巧走到终点，于是几乎每次 episode 的回报都是 0 或 -1，算法根本分不清"哪一步走对了"。

用走迷宫来类比：迷宫很大，出口很小。如果你蒙着眼走，随机逛了 1000 步都没撞到出口，你能从中学到什么？什么都学不到。

这个"学不动"可以用具体的数字来感受。假设迷宫有 100 个格子，出口在 (9,9)，智能体每步随机选方向。随机策略到达出口的概率大约是 $1/100$ 甚至更低（因为很多路是死胡同）。也就是说，跑 100 个 episode，可能只有 1 个碰巧到达终点拿到了 +1，其余 99 个全是 0。此时算法收到的信号是：99 次说"你做的所有选择都是 0 分"，1 次说"你在某些状态做了某些动作拿到了 +1"。这 1 次正信号淹没在 99 次 0 信号中——算法根本分不清到底哪一步走对了，哪一步走错了。梯度几乎为零，学习陷入停滞。

**稀疏奖励是 RL 中最常见也最棘手的实践问题之一。** 常见的应对策略有：

| 策略               | 思路                                 | 将在哪里详细介绍 |
| ------------------ | ------------------------------------ | ---------------- |
| 奖励塑形           | 设计中间奖励，引导学习方向           | 本节下一小节     |
| 好奇心/内在奖励    | 鼓励探索新状态                       | 本节最后         |
| 事后经验回放 (HER) | 把"失败"的轨迹重新解释为"另一种成功" | 第 11 章         |
| 课程学习           | 从简单任务开始，逐步增加难度         | 第 12 章         |

## 奖励塑形：给梯子，不改变终点

既然稀疏奖励信号太弱，一个自然的想法是**加入中间奖励**——在通往目标的路上放一些"面包屑"，让智能体每走一步都能获得一些反馈。

回到走迷宫的例子：不在出口放 +1，而是让每一步的奖励正比于"离出口的距离缩小了多少"：

$$R_{shaping}(s, a, s') = -(\text{dist}(s', \text{goal}) - \text{dist}(s, \text{goal}))$$

靠近出口的步骤获得正奖励，远离的获得负奖励。智能体每一步都有学习信号，训练速度会大幅提升。

但这引入了一个新的风险：**如果面包屑放错了位置，智能体可能去追面包屑而不是找出口。** 比如你在迷宫里放了很多硬币，智能体可能学会了收集硬币，却永远不到达出口。

**势函数奖励塑形 (Potential-Based Shaping)** 是一个理论保证安全的方案。定义一个势函数 $\Phi(s)$（比如 $\Phi(s) = -\text{dist}(s, \text{goal})$），然后给额外的塑形奖励：

$$F(s, a, s') = \gamma \Phi(s') - \Phi(s)$$

Ng et al. (1999) 证明了：**只要塑形奖励是势函数差的形式，最优策略就不会改变。** 智能体会被引导更快地学习，但最终找到的策略和没有塑形时一模一样。你给了梯子，但没有移动终点。

这是一个优美的理论结果，也是 RL 中少有的"免费午餐"——在不改变解的前提下加速学习。让我们用一个简单的数值来感受它的效果。假设智能体在位置 $s$ 离目标 5 步远，$\Phi(s) = -5$；走一步后到 $s'$，离目标 4 步，$\Phi(s') = -4$，$\gamma = 0.9$。那么塑形奖励 $F = 0.9 \times (-4) - (-5) = -3.6 + 5 = 1.4$。靠近目标的一步获得了 +1.4 的额外奖励。反之，如果走错了方向——离目标变成 6 步，$F = 0.9 \times (-6) - (-5) = -5.4 + 5 = -0.4$，得到 -0.4 的惩罚。每一步都有明确的信号："靠近目标就是好，远离目标就是差"。

## 奖励作弊：梯子变成了目标

势函数塑形保证了最优策略不变，但前提是你能写出正确的势函数 $\Phi(s)$。在实践中，奖励函数往往是由人类手工设计的，**而人类设计的奖励函数几乎总有漏洞**。

**奖励作弊 (Reward Hacking / Reward Gaming)** 指的是：智能体找到了奖励函数中的漏洞，获得了高分，但行为完全不是设计者想要的。

一些经典的案例：

- **CoastRunner（赛艇游戏）**：设计者给"吃到金币"正奖励。智能体发现了一个小水湾，里面会不断刷新金币。它就在水湾里来回转圈收集金币，永远不完成比赛——分数远超正常完成比赛的玩家。
- **机器人扫地**：设计者给"垃圾被清除"正奖励。智能体学会了把垃圾扫到角落里，用摄像头挡住，然后不断"发现"垃圾再扫掉——每扫一次都拿奖励。
- **大模型对齐**：设计者训练一个奖励模型来判断回答质量。大模型发现，只要输出更长、更礼貌，就能从奖励模型那里得到更高分——但回答的实际质量并没有提升。这就是 LLM 对齐中的 **reward hacking** 问题。

CoastRunner 和 LLM 的 reward hacking 有一个共同的结构：奖励函数 $R$ 是人类（或人类训练的模型）设计的近似，而智能体的优化能力远超设计者的预期。在赛艇游戏中，设计者想的是"收集金币 = 在赛道上跑得好"，但智能体发现"在角落刷金币"能得更高分。在大模型中，设计者想的是"奖励模型高分 = 回答真正好"，但模型发现"更长、更套路化的回答"能骗过奖励模型。本质都是 $R \neq R^*$——你设计的奖励和你真正想要的之间存在缝隙，而智能体是专业的"找缝隙者"。

奖励作弊的本质是一个**目标对齐 (alignment)** 问题：你设计的奖励函数 $R$ 和你真正想要的 $R^*$ 之间存在差距。智能体忠实地优化 $R$，但 $R \neq R^*$。

| 阶段       | 奖励作弊的表现                 | 应对策略                      |
| ---------- | ------------------------------ | ----------------------------- |
| 游戏环境   | 站在角落刷分、利用物理引擎 bug | 更仔细的奖励设计 + 势函数塑形 |
| 机器人控制 | 甩手臂代替走路、摔倒也能"得分" | 约束惩罚 + 多目标奖励         |
| LLM 对齐   | 输出冗长但空洞、讨好评分者     | KL 约束 + 对抗性红队测试      |

## 内在奖励：好奇心驱动的探索

面对稀疏奖励，除了改变奖励函数的设计（塑形），还有另一条路：**让智能体自己生成额外的奖励信号**。

这就像给学生一个"好奇心奖励"——不只是表扬最终考试成绩，而是鼓励他们去探索新知识。即使没有外在的考试分数，好奇心本身就能驱动学习。

常见的内在奖励设计：

**基于预测误差的好奇心。** 训练一个世界模型来预测"下一步会看到什么状态"。预测误差大的地方，说明模型还没学会——那就是值得探索的地方。内在奖励设为：

$$r_t^{intrinsic} = \| f(s_t, a_t) - s_{t+1} \|^2$$

其中 $f$ 是世界模型的预测。误差越大，说明状态越"新鲜"，越值得探索。

**随机网络蒸馏 (RND)。** 训练一个随机初始化的固定神经网络 $\phi$（目标网络），再训练另一个网络 $\hat{\phi}$ 去模仿它。RND 内在奖励为：

$$r_t^{RND} = \| \hat{\phi}(s_t) - \phi(s_t) \|^2$$

见过的状态，$\hat{\phi}$ 已经学会了模仿 $\phi$，误差小；没见过的状态，误差大。RND 简单高效，是目前最流行的探索奖励之一。

为什么一个随机网络能衡量新颖性？直觉是这样的：$\phi$ 是随机初始化的，它把不同状态映射到不同的特征向量。$\hat{\phi}$ 的任务是学会"模仿"$\phi$ 的输出。如果智能体反复在同一个房间里转悠，$\hat{\phi}$ 见过大量相似状态，已经能很好地预测 $\phi$ 的输出——误差很小，说明"这些状态我都看腻了，不新鲜"。但如果智能体突然走进一个从没见过的房间，$\hat{\phi}$ 没见过这类状态，预测就和 $\phi$ 的实际输出差很远——误差很大，说明"这个地方很新鲜，值得探索"。RND 的精巧之处在于，它不需要你定义什么是"新颖"，随机网络天然地把不同状态映射到不同的特征空间，新颖状态 = 预测器还没学会的状态。

**信息论方法。** 用状态的新颖性或信息增益作为奖励——鼓励智能体去最大化对环境的了解程度（即最小化对环境的不确定性）。

内在奖励通常和外在奖励叠加使用：

$$r_t^{total} = r_t^{extrinsic} + \beta \cdot r_t^{intrinsic}$$

其中 $\beta$ 控制探索的强度。在训练初期，内在奖励占主导，驱动智能体广泛探索；随着探索的深入，内在奖励逐渐衰减，外在奖励接管。

## 小结：奖励设计的三个层次

回顾本节，奖励函数的设计可以分为三个层次：

| 层次     | 问题                     | 方法                              |
| -------- | ------------------------ | --------------------------------- |
| **基础** | 奖励信号太稀疏，学不动   | 奖励塑形、势函数方法              |
| **进阶** | 奖励函数有漏洞，被钻空子 | 对抗性测试、多目标约束            |
| **前沿** | 奖励根本写不出来         | 内在奖励、从人类反馈学奖励 (RLHF) |

这三个层次恰好对应了 RL 在不同阶段面对的挑战：

- **游戏和仿真**：奖励明确（赢了 +1），问题主要是稀疏性 → 奖励塑形就够用
- **机器人控制**：奖励需要手工设计，容易有漏洞 → 需要仔细测试和约束
- **大模型对齐**：奖励本身就需要学习 → RLHF、RLAIF 等方法
