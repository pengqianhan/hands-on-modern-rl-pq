# 第8章：算法选型指南

> **本章目标**：帮助你在面对实际问题时，快速选择合适的 RL 算法。不是追求"最强算法"，而是找到"最合适的算法"。

## 8.1 决策速查表

| 场景 | 推荐算法 | 核心理由 | 不推荐 |
|------|---------|---------|--------|
| 游戏控制（离散动作） | PPO / DQN | 环境反馈密集，样本充足 | REINFORCE（方差太大） |
| 游戏控制（连续动作） | PPO / TD3 / SAC | 需要精确的连续控制 | DQN（无法处理连续空间） |
| 小模型快速对齐 | DPO | 实现简单，显存友好 | PPO（杀鸡用牛刀） |
| 推理增强（数学/代码） | GRPO | 规则奖励可验证，Critic 可省 | DPO（缺乏在线探索） |
| 复杂人类偏好对齐 | PPO | 奖励模型精细，上限更高 | DPO（受静态数据限制） |
| 多模态 RL | GRPO / PPO | 需要组内比较处理异构输入 | DPO（数据标注成本高） |

## 8.2 按维度选择的思考框架

- **动作空间**：离散 → 优先考虑 DQN/PPO；连续 → 优先考虑 PPO/SAC/TD3
- **奖励信号**：密集 → PPO/DQN 都行；稀疏 → 需要奖励塑形或好奇心驱动
- **样本效率**：on-policy（PPO）效率低但稳定；off-policy（SAC/TD3）效率高但工程复杂
- **在线 vs 离线**：能在线采样 → PPO/GRPO；只有历史数据 → 离线 RL 或 DPO
- **计算资源**：显存充足 → PPO 全套；显存紧张 → GRPO 或 DPO

## 8.3 常见错误

- 用 DQN 处理连续动作空间（DQN 只支持离散动作）
- 在数据量不足时用 PPO（on-policy 需要大量采样）
- 用 DPO 做需要在线探索的任务（DPO 只能用固定的偏好数据集）
- 忽视环境特性（确定性 vs 随机性、稀疏奖励 vs 密集奖励）直接选"最强算法"
- **你将获得**：避免"什么都用 PPO"或"什么都用 DPO"的选型陷阱
